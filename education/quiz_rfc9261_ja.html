Q1: RFC 9261 は主に何を解決するための仕様か

Multiple Choice
**Explanation:** **問題を出した背景:** RFC 9261 は certificate の新形式ではありません. 中心は, すでにある TLS または DTLS connection に結びついた **post-handshake identity proof** です. **用語:** **Exported Authenticator** は, X.509 certificate などの identity を peer が本当に扱えることを示す proof です. その proof は initial connection から exporter で導かれる材料に結びつきます. **実務での機会:** 1 本の secure connection の上で, あとから別の名前, tenant, role, application event に対する証明が必要になる設計で出てきます. **選択肢:** - A (incorrect): X.509 を置き換える話ではありません. - B (correct): RFC 9261 の主目的を捉えています. - C (incorrect): HTTP/3 header compression は QPACK の領域です. **関連:** RFC 9261 は TLS handshake 後の追加認証を application protocol が扱うための部品として読むと理解しやすいです.

Q2: Exported Authenticator が必要になる背景として正しいものはどれか. 複数選択

Multi-Select
**Explanation:** **問題を出した背景:** RFC 9261 の introduction は, **multiple identities** と **spontaneous authentication** を大きな動機として挙げています. **用語:** **multiple identities** は, handshake で示した identity 以外もあとから証明したい状況です. **spontaneous authentication** は, application 側のイベントに応じて認証を発生させることです. **実務での機会:** multiplexed protocol や tenant-aware な service では, 接続開始時ではなく操作ごとに証明が必要になることがあります. **選択肢:** - A (correct): 複数 identity の動機に合っています. - B (incorrect): certificate validation を省くための仕組みではありません. - C (correct): application event に応じた追加認証です. - D (incorrect): DNSSEC の話ではありません. **関連:** TLS record boundary と application message boundary が揃わない protocol では, この切り分けが特に重要になります.

Q3: request-based flow の形として正しいものはどれか

Multiple Choice
**Explanation:** **問題を出した背景:** Exported Authenticator を certificate blob とだけ見ると, request と validation の文脈を落とします. **用語:** **authenticator request** はどんな proof を求めるかを定義します. **authenticator** は certificate に関わる proof を持ち, 受け取った peer が検証します. **実務での機会:** application protocol を設計するとき, request の運び方, authenticator の運び方, validation error の返し方を決める必要があります. **選択肢:** - A (correct): request-based sequence を正しく表しています. - B (incorrect): RFC 9261 は TLS layer の state machine を作り直す仕組みではありません. - C (incorrect): certificate chain だけでは, その connection context で private key を持つ証明になりません. **関連:** server は request なしで spontaneous authenticator を出せる場合がありますが, client は先行 request なしに authenticator を送れません.

Q4: TLS 1.3 の post-handshake authentication と比べたとき, RFC 9261 の特徴はどれか

Multiple Choice
**Explanation:** **問題を出した背景:** RFC 9261 は renegotiation や TLS 1.3 post-handshake authentication の限界を背景にしています. **用語:** **post-handshake authentication** は handshake 後の認証です. **TLS state machine** は TLS layer が内部で持つ protocol 状態です. **実務での機会:** HTTP/2 のような multiplexed protocol では, TLS record boundary と application message boundary が一致しません. そのため application 側で扱える proof のほうが設計しやすい場合があります. **選択肢:** - A (incorrect): RFC 9261 は connection 確立後の proof を扱います. - B (incorrect): support negotiation や error handling は application 側の責務です. - C (correct): これが設計上の重要な違いです. **関連:** Exported Authenticator は完成した protocol ではなく, protocol を作るための building block です.

Q5: authenticator は大きく見ると何から構成されるか

Multiple Choice
**Explanation:** **問題を出した背景:** authenticator の構造は TLS 1.3 の認証パターンに似せています. これはすでに分析されている形を再利用するためです. **用語:** **Certificate** は identity を運びます. **CertificateVerify** は対応する private key の所有を示します. **Finished** は exporter-derived な材料を使って authenticator transcript に MAC をかけます. **実務での機会:** TLS API の返すものが単なる certificate chain なのか, connection に結びついた authenticator なのかを review するときに効きます. **選択肢:** - A (incorrect): private key を送ることはありません. - B (correct): RFC 9261 の high-level construction です. - C (incorrect): TLS record の保管や圧縮ではありません. **関連:** authenticator は TLS handshake message として encode されますが, TLS record-layer framing は含みません.

Q6: certificate_request_context について覚えるべきことはどれか. 複数選択

Multi-Select
**Explanation:** **問題を出した背景:** **certificate_request_context** は小さな field ですが, replay や context confusion を防ぐうえで重要です. **用語:** **certificate_request_context** は request を識別する opaque value です. **opaque** は protocol が中身を名前として解釈しない byte string という意味です. **実務での機会:** API を作るとき, context の生成, 保存, 使用済み管理を雑にすると validation の意味が崩れます. **選択肢:** - A (correct): request と authenticator を結びつけます. - B (incorrect): 同じ connection 上での再利用を避けるため uniqueness が必要です. - C (incorrect): CA が発行する subject name ではありません. - D (correct): uniqueness は要求で, unpredictability は推奨です. **関連:** nonce と似ています. 値そのものより lifecycle の扱いが security property を支えます.

Q7: validation における connection binding とは何を意味するか

Multiple Choice
**Explanation:** **問題を出した背景:** Exported Authenticator は持ち運び可能な certificate receipt ではありません. initial TLS または DTLS connection から exporter で導いた材料に結びつきます. **用語:** **connection binding** は proof が特定 connection の secret に結びつくことです. **CertificateVerify** は exported Handshake Context を含む transcript に署名します. **実務での機会:** TLS-terminating proxy が入る構成で表面化します. application が見ている接続と validation が参照する接続がずれると proof は合いません. **選択肢:** - A (incorrect): 同じ certificate だけでは不十分です. - B (incorrect): DNS name validation は必要でも, proof のすべてではありません. - C (correct): connection secret が違えば Handshake Context も違い, validation は失敗します. **関連:** validation 失敗時は, 両者が同じ underlying TLS connection を見ているかを確認する必要があります.

Q8: empty authenticator は何のためにあるか

Multiple Choice
**Explanation:** **問題を出した背景:** refusal でも integrity が必要なことがあります. RFC 9261 は **empty authenticator** で, 単なる無応答とは違う拒否を表現します. **用語:** **empty authenticator** は Certificate や CertificateVerify なしの Finished message です. refusal は認証されますが, identity は認証されません. **実務での機会:** application UX や error handling で, well-formed refusal と壊れた response を分けたいときに役立ちます. **選択肢:** - A (correct): これが intended role です. - B (incorrect): well-formed empty authenticator は identity validation としては invalid です. - C (incorrect): TLS version upgrade signal ではありません. **関連:** empty authenticator は protocol flow の一部であり, requested identity の存在証明ではありません.

Q9: application protocol 側に残る責務はどれか. 複数選択

Multi-Select
**Explanation:** **問題を出した背景:** RFC 9261 は building block です. これを使う application protocol にはまだ設計すべき部分があります. **用語:** **application protocol** は TLS または DTLS の上の層です. message format, capability signaling, user-visible behavior を決めます. **実務での機会:** protocol extension を設計するとき, support discovery, message transport, failure semantics を決めずに暗号部品だけ入れても動きません. **選択肢:** - A (correct): support advertisement は TLS が自動でやってくれるわけではありません. - B (correct): request と authenticator は application layer で運べます. - C (incorrect): certificate-chain validation は依然として必要です. - D (correct): error handling は application protocol policy です. **関連:** TLS library は cryptographic API を提供できますが, 周囲の契約は application が定義します.

Q10: RFC 9261 の validation と API guardrail として重要なものはどれか. 複数選択

Multi-Select
**Explanation:** **問題を出した背景:** RFC 9261 は wire format だけでなく, 壊れにくい API を作るための実装上の guardrail も示しています. **用語:** **extended master secret** は TLS 1.2 の key synchronization problem を避けるために重要です. **constant-time comparison** は timing leakage を減らす検証方法です. **実務での機会:** TLS library に Exported Authenticator API を持たせるか, exporter だけを application に渡すかを検討するときに出ます. **選択肢:** - A (correct): TLS または DTLS 1.1 以前では fail すべきです. - B (incorrect): TLS 1.2 では extended master secret が必要です. - C (correct): signature と MAC validation では constant-time comparison が推奨されています. - D (correct): RFC 9261 は TLS library 内での実装を推奨しています. **関連:** library と application の境界は意図的に設計すべきです. TLS library は certificate や extension selection を扱いやすく, application は protocol semantics を扱います.
